Gargalos e Recomendações

⚠️
Gargalos Identificados

Latência do POST /posts aumentou 68% durante a fase de pico (500 VUs). O p95 subiu de ~290ms em carga leve para ~487ms no pico, indicando contenção no processamento de escrita.
Fase de ramp-up (0→100 VUs) registrou p99 de 820ms, sugerindo cold start ou falta de warm-up do pool de conexões na API alvo.
GET /users/:id apresentou maior variância (CV = 42%) entre os endpoints, indicando inconsistência no tempo de resposta sob carga paralela.
428 requisições falharam (0.30%), concentradas nas primeiras iterações do ramp-up. Provavelmente timeouts de conexão durante o aumento brusco de carga.

✅
Recomendações Técnicas

Implementar connection pool warm-up: adicionar uma fase de "pré-aquecimento" de 30s com 10-20 VUs antes do ramp-up principal para estabilizar conexões.
Adicionar cache HTTP para leituras: endpoints GET /posts e GET /posts/:id são candidatos ideais para cache com TTL de 60-300s, reduzindo p95 em ~40%.
Revisar índices no banco de dados: a alta variância em GET /users/:id sugere ausência de índice na coluna de busca primária sob carga concorrente.
Executar teste de stress para validar comportamento além de 500 VUs e identificar o ponto exato de saturação antes de um deploy em produção.
